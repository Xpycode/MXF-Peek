Build + test cycle hit a non-obvious bug that took three attempts to nail: URL equality assertions against the scanner's output kept failing while count-only assertions passed. Diagnosis landed via xcresult JSON (not in stdout): `row.fileURL → file:///private/var/folders/...` vs test-side `file:///var/folders/...`. Root cause: `/var` on macOS 10.15+ is an **APFS firmlink**, not a traditional symlink. `URL.resolvingSymlinksInPath()` is a no-op on firmlinks by design; `FileManager`'s enumerator canonicalizes through the firmlink so its URLs come out with the resolved prefix. Fix: `URLResourceValues.canonicalPath` on the *parent* (which exists and can be stat'd), then `appendingPathComponent` the UUID-suffixed leaf. `realpath(3)` and `canonicalPath` both resolve firmlinks; `resolvingSymlinksInPath()` and `URL.standardized` do not. ### Framework landscape survey
Discussed Windows desktop app options with a focus on stability and longevity (not just newest):

- **WPF (.NET)** — 20 years old, open-source since .NET Core 3.0, ships with .NET, massive third-party control ecosystem (Telerik/Syncfusion/DevExpress). Best fit for a data-heavy auditor app. The clear recommendation.
- **WinForms** — even older, zero-friction for utility-grade UI, maintained in .NET. Fine if UI stays simple.
- **Qt (C++ / PyQt)** — 30+ years, commercially backed, genuinely cross-platform. Real option if a cross-platform codebase is ever desired.
- **WinUI 3** — Microsoft's stated long-term successor to UWP. Stabilizing but smaller community than WPF today.
- **.NET MAUI** — cross-platform (Win/Mac/iOS/Android). Still maturing; abstraction cost is real.
- **Electron / Tauri** — web-tech wrappers. Valid but heavy (Electron) or newer community (Tauri).

Clarified the historical "please install .NET" problem vs. the current situation. Modern .NET supports **self-contained single-file publish** — bundles the entire runtime inside the app. User installs nothing. Size penalty ~60–80 MB. # Session: 2026-04-21 (late) — Player design revision + implementation plan

## Goal
Revise the player approach from "cached full-file transcode" to a smarter design, then produce a proper implementation plan ready to execute in the next session.

## Actions

### Design revision based on user pushback

The spec drafted earlier proposed a simple cached-transcode model: ffmpeg remuxes/transcodes the MXF → writes a temp `.mov` → AVPlayer opens the file. Startup delay was linear in clip length (1 s for a 1-min clip, 40 s for 13 min, ~2 min for the 38-min Kowloon outlier).

User pushed back on two points that both held up:

1. **UNC-path fast-path was semantically broken** for an audit tool. `comment_UNC Path` points to the *pre-Avid* source file (YouTube mp4, Premiere export). Playing that instead of the MXF means the audit is lying about what it's auditing. Scrapped.

2. **"What about live-muxing? What about a player that plays 1 video + up to 8 audio streams and combines them? NLEs can do that."** Correct on both counts; I'd under-weighted live-mux and under-scoped what NLEs actually do. Did a proper research pass on the 7 viable approaches.

### Research pass (ultrathink mode)

Web + Apple docs searches, targeted probes:

- **HLS via `file://`:** doesn't work in AVPlayer ([Apple forums 69357](https://developer.apple.com/forums/thread/69357)). HLS is a network protocol in AVFoundation's design.
- **`AVAssetResourceLoaderDelegate` for HLS:** can intercept the `.m3u8` playlist but **not** the segment files — Apple reserves segment loading for its internal bitrate-adaptation logic ([Apple forums 113063](https://developer.apple.com/forums/thread/113063); [Jared Sinclair, 2016](https://jaredsinclair.com/2016/09/03/implementing-avassetresourceload.html)). So HLS requires a real HTTP endpoint. The "server" isn't about permissions, it's about a specific API constraint.
- **Loopback HTTP server UX:** invisible. Binding to `127.0.0.1` via `NWListener.requiredLocalEndpoint = .hostPort(host: .ipv4(.loopback), port: .any)` doesn't trigger the Local Network prompt (multicast/Bonjour-only), doesn't trigger the firewall (loopback bypasses it), needs no entitlements.
- **Fragmented MP4 progressive playback (no server):** theoretically works but has multiple broken-playback reports on growing files ([Apple forums 744598](https://developer.apple.com/forums/thread/744598)); not production-safe without heavy validation.
- **`AVSampleBufferDisplayLayer` + ffmpeg-as-library:** the pure NLE architecture. ~500-800 LOC. Free audio-pair switching (just toggle `AVAudioPlayerNode`s). v2+ ambition.
- **KSPlayer** (Swift package, AVPlayer+FFmpeg wrapper): GPL license by default, LGPL version is paid → **ruled out for a distributable app**.
- **VLCKit**: plays MXF via libvlc but multi-stem sync via `--input-slave` is [flagged experimental and broken](https://forum.videolan.org/viewtopic.php?t=73488). Pays the bundle cost without solving the hard part. Ruled out.

**Option matrix captured in the research reply** with seven approaches scored on startup latency, disk cost, code complexity, and audio-pair switching support.

### Chose Option 2: HLS live-mux + localhost loopback HTTP server + AVPlayer

**Why:**
1. Uniform ~0.5–2 s startup regardless of clip length (solves the 38-min Kowloon pain Option 1 cannot).
2. `AVMediaSelectionGroup` swaps audio pairs natively — one transcode, instant switch.
3. Server is invisible to the user (loopback-only, no prompts/creds/ports surfaced).
4. Only ~300 LOC over Option 1's ~150 LOC.

Option 5 (custom NLE player) deferred as v2 ambition — worth revisiting after v1.2 ships.

### Two de-risking probes before committing to the plan

1. **ffmpeg HLS command against real user MXF.** First attempt failed (`-map "0:v"` with quotes, DNxHR encode constraints). Second attempt succeeded: `-filter_complex "[1:a:0][2:a:0]join=inputs=2:channel_layout=stereo[pair0]" -map 0:v:0 -c:v h264_videotoolbox -b:v 4M -pix_fmt yuv420p -map "[pair0]" -c:a aac -b:a 192k -f hls -hls_time 4 -hls_segment_type fmp4`. Output: 30 s of valid HLS (init.mp4 + 8 fMP4 segments + playlist.m3u8) in 3.3 s wall time via VideoToolbox hardware encode (~9× realtime). Key insight: **encode beats realtime, so first segment lands in <1 s wall time → AVPlayer can open the playlist immediately.**

2. **NWListener loopback binding syntax.** Confirmed via search: `params.requiredLocalEndpoint = NWEndpoint.hostPort(host: .ipv4(.loopback), port: .any)` is the clean form. Kernel-assigned port read from `listener.port` after `.ready` state.

### Implementation plan written

`docs/plans/2026-04-22-player-hls.md` — 8 waves:
- **P1** De-risking spike (4 h, timeboxed) — answers remaining open questions before the full build commits. Exit criteria: live-mux startup < 5 s verified, `AVMediaSelectionGroup` audio switch verified, full-range scrubbing verified.
- **P2** ffmpeg bundling — rename `bundle-ffprobe.sh` → `bundle-toolchain.sh`, extend signing, add `.ffmpeg` to `BundledTool` enum. Bundle 74 MB → ~165 MB.
- **P3** `PreviewHTTPServer` (~200 LOC actor) — NWListener, loopback-only, HTTP/1.1 GET, Content-Type by extension, optional range support.
- **P4** `PreviewTranscoder` (~250 LOC) — ffmpeg subprocess with `-progress pipe:1` reading, cancellation via `Process.terminate`, `FSEvents` for `firstSegmentReady` signal. Plus `AudioPair` model with filename-convention pairing.
- **P5** `PreviewCache` (~180 LOC actor) — SHA-256 hash keying on `materialKey + file paths/sizes`, 10 GB LRU eviction, disk preflight.
- **P6** `PlaybackCoordinator` + `PlaybackState` (~230 LOC) — observes ScanModel selection, wires transcoder events to `@Observable` state.
- **P7** `PlayerView` (~150 LOC) — `AVPlayerView` via NSViewRepresentable, audio-pair Picker, preparing/failed overlays, integration into `ClipInspectorView`.
- **P8** Adversarial + polish — mirrors 5.3 (missing stem, corrupt stem, 0-audio, odd stems, disk full, rapid clip switching, zombie-ffmpeg at quit).

Total new Swift: ~1,620 LOC + 90 MB ffmpeg binary. Full build after spike: ~18–22 h across 3–4 sessions. Plan doc has verified ffmpeg command, NWListener syntax, architecture diagram, risk register, and file inventory.

## Decisions

- **Option 2 (HLS live-mux + loopback server) chosen over Option 1 (cached full transcode).** 38-min clips can't be gated on 2-minute startup. Uniform ~1-2 s startup via live-mux is the correct UX.
- **UNC-path fast-path removed from the design entirely.** Audit tool integrity requires showing what's in the MXF, not what was in the pre-Avid source.
- **H.264 via VideoToolbox** chosen as the proxy codec (over DNxHR LB). Smaller output (~15 MB/min vs ~280 MB/min), hardware-accelerated, universally playable in AVPlayer, quality acceptable for preview.
- **HLS playlist type = `event` + `append_list`** (not `vod`) — growing playlist enables live-mux. Verified by probe that playlist updates incrementally.
- **Plan stays in draft until P1 spike lands.** Same discipline as spec — don't commit multi-wave implementation to unresolved risks.
