### §10 post-spike plan corrections (MCP + web research)
After spike, ran an ultrathink research pass using apple-docs MCP, codebase audit, and targeted web searches (RFC 7233, HLS forum threads, ffmpeg -progress docs, DispatchSource gotchas). Surfaced **5 plan drifts** between the original §2–§3 design and reality:

2. **§10.2** HTTP `Range:` support is **required**, not optional. Per Apple forums, AVPlayer uses byte-range requests for HLS duration calculation during init. Server must emit `206 Partial Content` + `Content-Range`, `416` for out-of-bounds, `501` for multi-range.
3. **§10.3** Cache math 2× optimistic: 4 Mbps + 192 kbps AAC = **30 MB/min, not 15**. 10 GB = ~33 clip-min. Revise budget/bitrate at Wave P5.
4. **§10.4** `firstSegmentReady` via 100 ms poll, not `DispatchSource.makeFileSystemObjectSource` — the dispatch-source API monitors single fds; watching a directory for new files needs re-enumeration anyway.
5. **§10.5** New risk row: **first-play seek past transcode horizon** may stall AVPlayer. Cache-hit replays unaffected (ENDLIST present). Needs P8.2 manual QA on a 30-min clip.

### Wave P3 — PreviewHTTPServer (commit `37a7bb5`)
`Services/PreviewHTTPServer.swift` ~306 LOC — actor wrapping `NWListener`. Loopback-only via `requiredLocalEndpoint = .hostPort(.ipv4(.loopback), .any)`, dynamic port, GET-only. **Range support per §10.2** — single-range `bytes=START-END`/`bytes=START-` → 206 + `Content-Range`, out-of-bounds → 416 + `bytes */TOTAL`, multi-range → 501. Path-traversal guard via `standardizedFileURL.hasPrefix(rootDir + "/")`. 16 KB header cap.

**Key architecture choice:** connection serving is `nonisolated static` (only reads the `nonisolated let rootDir`), so concurrent requests don't serialize through the actor. Only listener lifecycle + `baseURL` touch actor-isolated state. Kept `Connection: close` per response — AVPlayer opens parallel connections anyway, keep-alive adds complexity without benefit for static read-only workload.

11 tests all green — start/stop lifecycle, 404, 403 (path traversal via `/..%2F`), MIME dispatch, Range 206/416/501, 10 concurrent GETs in 0.030 s, idempotent start, stop-and-restart.

### Wave P5 — PreviewCache (commit `62eb143`)
`Services/PreviewCache.swift` ~250 LOC — actor. `pathIfCached` / `prepareOutputDir` / `markComplete` / `markFailed` / `evictToFit` / `totalSize` / `state(for:)` / `defaultRootDir()`. Hash = SHA-256 of `"\(materialKey)|\(path1)|\(size1)|…"` (sorted files), first 16 hex chars. `.transcode-state` JSON sidecar: `{status, pid, startedAt, accessedAt, reason}`, ISO8601 dates, pretty-printed. LRU sort by `accessedAt` asc; **`.running` entries never evicted**. Preflight threshold default 1 GB, overridable for tests.

14 tests — 4 hashing (stable/order-indep/changes-on-size/path/materialKey), 3 prepareOutputDir (creates+running, wipes prior, throws-on-preflight), 4 pathIfCached (missing/running/failed/complete), 3 eviction (no-op under budget, removes oldest first, protects running).

**Bug caught during development:** `URL.appendingPathComponent(_:)` **without explicit `isDirectory:`** does a filesystem probe to set `hasDirectoryPath`. The flag flips between calls as the directory gets created, which breaks URL equality across `prepareOutputDir` invocations on the same clip (first call returns URL with `hasDirectoryPath=false`, second with `=true`). Fix: always `isDirectory: true` for dir paths. Probably cookbook-worthy.

**Eviction test authoring gotcha:** initial attempt with `Task.sleep(nanoseconds: 20_000_000)` between `markComplete` calls was flaky under parallel-test load — fixed by writing the `.transcode-state` file directly with explicit timestamps from a fixed base date. Removes all timing dependencies.

## Decisions
- **Wave P1 spike exit criteria all green** → v1.2 player design promoted from "maybe viable" to confirmed-shippable. ~2.5 s uniform startup latency is the killer feature vs Option 1 (cached full-file transcode was linear in clip length).
- **HTTP Range support is not optional** (§10.2): promoted from "skip if P1.7 doesn't need it" to mandatory 206/416/501 handling in `PreviewHTTPServer`. AVPlayer uses byte-ranges for duration probing before segment fetching.
- **Cache LRU sort key is `accessedAt` from the JSON sidecar, not the filesystem atime.** More portable (APFS atime behavior varies), explicit, and easy to unit-test by writing state files directly.
- **`.running` cache entries are immune from eviction.** Separate concept from LRU — in-flight transcodes we don't want to interfere with, even if they're older than completed ones.
- **Chose struct for PreviewTranscoder over actor.** No shared mutable state across transcode calls; each returned AsyncStream closes over its own Process. Simpler, no actor-hop overhead for the stream consumer.
- **Process termination uses `terminate()` then SIGKILL via `DispatchQueue.asyncAfter(2.0)`.** Two-phase cancellation gives ffmpeg a chance to flush HLS footer cleanly before the hard kill.
- **pbxproj main target stays on legacy PBXGroup.** Not migrating to synchronized folder this session — the per-file registration (4 edits each) is mechanical and reliable, and the test target's synchronized folder already handles the high-churn test files.
- **Spike artifacts deleted after validation**, findings persisted to plan doc §10 + session log. No throwaway code kept in-tree.
