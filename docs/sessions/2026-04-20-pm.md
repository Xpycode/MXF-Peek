## Wave 2 — parse & scan (completed same session)

### 2.1 `MXFHeaderInfo` — done, co-located in `BMXWrapper.swift`
Sendable, Identifiable struct. Fields: fileURL, fileSize, materialPackageUID, filePackageUID, editRateNum/Den, durationFrames, videoTrackCount, audioTrackCount, clipName, projectName, tapeName, parseError, parseDurationMs. Derived: trackCount, durationSeconds, shortMaterialUID (16-char tail), displayName (clipName ?? filename stem). Static `.failed(url:size:reason:)` constructor for the error row shape.

**Placement decision:** inline in `Services/BMXWrapper.swift` rather than its own file — avoids pbxproj surgery for a new file reference. Split out into `Models/MXFHeaderInfo.swift` whenever we next need to touch pbxproj anyway.

### 2.2 `BMXWrapper.info(url:)` — done
`func info(url: URL) async -> MXFHeaderInfo` — does NOT throw for per-file errors, returns `.failed(...)` instead so a 500-file scan never stalls on one bad clip. Only async-throwing path is process-launch failure. Added private helper `runMxf2Raw(at:arguments:) async throws -> Data` that's separate from `runBMX` — no progress/log plumbing, just fire-and-collect stdout. Added internal `DataCollector` class (`@unchecked Sendable` via NSLock) for pipe accumulation. Invocation: `mxf2raw --info --info-format xml --avid <url>`. Library path injected via existing `toolResolver.bmxEnvironment()`.

**XML parsing:** new `enum MXFInfoXMLParser` with `static func parse(...)` uses `XMLDocument` (Foundation on macOS). Strategy: walk all descendants, collect leaf-element text into a flat `[localName: String]` map with first-occurrence-wins, then pick fields by trying several known key spellings (`material_package_uid` / `materialpackageuid` / `material_package_umid` etc.). Track counting reads `type`/`kind`/`essence_kind` attributes on `<track>` elements, matching "picture"/"video" → video, "sound"/"audio" → audio. Schema knowledge is partial — revisit when Wave 5.2 provides real samples.

**Verified subprocess path** by running `mxf2raw --info --info-format xml --avid` against non-MXF files: errors go to stderr, exit code is non-zero, our pipeline captures stderr into `BMXError.conversionFailed(...)` and the scanner surfaces it as `MXFHeaderInfo.failed(reason:)`. Happy path is unverified until real MXFs arrive.

### 2.3 `MXFFolderScanner` — done, appended to `Services/P2CardParser.swift`
`struct MXFFolderScanner` with `maxConcurrent: Int = 8`. Three methods:
- `static discoverMXFFiles(under:) -> [URL]` — recursive enumeration, `.skipsHiddenFiles + .skipsPackageDescendants`, case-insensitive `.mxf` filter, sorted by path for stable order
- `scan(folder:) -> AsyncStream<MXFHeaderInfo>` — the real API. `Task.detached` wraps a `withTaskGroup` that seeds to concurrency cap, then drain-and-refill to stay near `maxConcurrent` wide. `continuation.onTermination` cancels the task when the stream is dropped. Each subtask makes a fresh `BMXWrapper()` (cheap — no shared cancel-state concern across parallel calls)
- `scanAll(folder:) async -> [MXFHeaderInfo]` — convenience collector for testing/small folders

**Placement decision:** appended to existing `Services/P2CardParser.swift` — the old `P2CardParser` class stays intact as dead code until pruning, coexisting with the new scanner. Same reason as 2.1 (avoid pbxproj surgery for new files).

## Decisions captured during Wave 2
- **Inline MXFHeaderInfo in BMXWrapper.swift** (vs. own file) — pbxproj-surgery avoidance outweighs textbook structure here. Move later as part of a batched pbxproj pass.
- **Scanner appended to P2CardParser.swift** — same rationale. Filename will rename to `MXFFolderScanner.swift` when we next do a pbxproj sweep.
- **`info(url:)` returns `MXFHeaderInfo` instead of `throws`** for per-file errors — scale-friendly. A bad MXF inside a 10k-file folder becomes one row with an error string, not a thrown error that kills the scan.
- **XMLDocument over XMLParser (SAX)** — we're macOS-only and XMLDocument's DOM-walking API is far less code than a SAX delegate. Trade-off: entire doc buffered in memory, but mxf2raw XML per file is small (<10 KB typically).

## Gotchas captured for future-me
- **`@unchecked Sendable` data collectors need a lock**, not "it works most of the time." P2toMXF's `OutputCollector` pattern reused here for `DataCollector` — NSLock around stringvar mutations.
- **`FileManager.enumerator` with `.skipsPackageDescendants`** matters for MXF scans — without it, the enumerator would recurse into `.photoslibrary`, `.app`, etc. Avid's `MediaFiles/MXF` hierarchy isn't packagey but defensive default avoids surprises on mixed folders.
- **`Task.detached(priority: .userInitiated)`** for the scan task — avoids inheriting actor/MainActor context from the caller. The AsyncStream can be consumed from anywhere; the scan work itself runs off the main thread.

Sendable, Identifiable, Hashable struct with `materialKey` (stable id — UMID or file URL string), `files: [MXFHeaderInfo]`, and `isUngroupable` flag. ### 3.2 `ClipAggregator` — done, same file
Pure `enum` with one `static func aggregate(_ infos: [MXFHeaderInfo]) -> [Clip]`. Groups by `materialPackageUID` (trimmed, non-empty); everything else becomes an ungroupable single-member clip keyed by `fileURL.absoluteString`. Output sorted by `displayName` using `localizedStandardCompare` so "A001_C010" sorts after "A001_C009" like humans expect.

### 3.3 `AuditReportExporter` — done, appended to `Services/ReportGenerator.swift`
`enum AuditReportExporter` with four entry points: `makeJSON`, `makeCSV`, `writeJSON(to:)`, `writeCSV(to:)`. Plus `jsonSchemaVersion = "1.0.0"` constant and a private `AuditReportDTO` struct for wire-format stability (explicit snake_case fields decoupled from internal model renames).

**CSV** — RFC 4180 escaping via `csvField(_:)` (quote if contains `,` `"` `\r` `\n`; internal quotes doubled); `\r\n` line endings; three metadata rows prefixed with `# ` above the column header (source_folder, generated_at, schema_version). 13 columns per clip.

**JSON** — pretty-printed, ISO 8601 dates, `.withoutEscapingSlashes`. Three nested DTO levels: top-level envelope → summary + clips; clip → files. Schema is version-gated; bump rules documented in the spec.

## Decisions captured during Wave 3
- **`Clip` and `ClipAggregator` live next to `MXFFolderScanner` in `P2CardParser.swift`.** All four types participate in the scan→aggregate→export pipeline; co-location is natural. File rename to `MXFFolderScanner.swift` batched with the Wave 4-boundary pbxproj sweep.
- **DTO for JSON encoding** rather than making `Clip`/`MXFHeaderInfo` Codable. Decouples wire format from internal model; schema version is the contract, not the struct layout. Additional boilerplate (~30 lines) is cheap insurance.
- **Metadata rows in CSV prefixed with `# `** — violates strict RFC 4180 (no concept of comments) but Numbers/Excel tolerate them; downstream scripts `grep -v '^#'`. Alternative (separate `.meta.json` sidecar) felt like overkill for a human-readable format. If a consumer complains, strip with the `--bare` flag when it's added.
- **Max duration vs. first duration** — used `.compactMap(\.durationFrames).max()` so a clip survives a partially-failed parse. For OP-Atom in practice all stems share duration, so max == first == all; the max form is just robust against the edge case where one stem's duration didn't parse.
- **`localizedStandardCompare` for sort** — handles `A001_C009` → `A001_C010` ordering that lexical sort would scramble. Same treatment Finder uses.

## Gotchas captured
- **Optional-chained tuple member access** (`clip.editRate?.num`) gives `Int32?`, not `Int32`. Tried adding an `.asOptional` extension "to make types clear" — pointless, produced `Int32??` at the call site. Removed the extension; direct access is already correct. Lesson: don't add helper extensions before reading the actual produced type.
- **`JSONEncoder.dateEncodingStrategy = .iso8601`** formats to seconds precision with `Z` suffix for UTC — matches what the schema doc promises. `.iso8601` in `outputFormatting` does NOT exist; that's a different concept. The DTO has `generated_at: Date` and the strategy does the formatting at encode time.
- **SourceKit timeouts on long mixed-expression array literals** are a real and persistent quirk. The threshold is surprisingly low (13 elements with mixed types + nil-coalesce + ternary was enough). `xcodebuild`'s production compiler has a higher budget and gets through it. For everyday edits SourceKit will be fine; only the first full-file re-analysis hits this.
