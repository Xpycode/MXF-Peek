### Wave P6 — PlaybackCoordinator + PlaybackState (commit pending)

Started by cross-verifying the plan's §P6 design against Apple docs (Observation framework + AVFoundation + AVKit) before writing code. Surfaced four pre-P6 corrections, appended as plan §11:

1. **§11.1** `withObservationTracking` fires once and detaches — re-arm inside `onChange` is load-bearing. Apple's only async-cleanup hook for non-SwiftUI observers; the `@Sendable` on the inner closure confirms it's meant to bridge to structured concurrency, not Combine. (No "Combine publisher for @Observable" exists.)
2. **§11.2** Task supersedence pattern made explicit — `currentPrepTask?.cancel()` before assigning, cooperates with the existing `AsyncStream.Continuation.onTermination` from P4 that already fires SIGTERM + 2s SIGKILL grace.
3. **§11.3** New P6.4 task: `AVPlayerItem.status` observer scoped inside `currentPrepTask` so it cancels with the prep cycle. Distinct from the §2.1 `firstSegmentReady` gate — the latter is "URL resolves on disk", the former is "AVFoundation has loaded the asset". Two different failure modes that share a `.failed` symptom.
4. **§11.4** `NSViewRepresentable(AVPlayerView)` over `SwiftUI.VideoPlayer` — reasoning upgraded from "we need a Picker" to "editorial workflow needs `showsTimecodes` + `showsFrameSteppingButtons` + JKL transport + `beginTrimming` + chapter-flash". `VideoPlayer` doesn't expose those.

Implementation:
- `Models/PlaybackState.swift` ~78 LOC — `@Observable` with `Phase { idle, preparing(startedAt, progress), playing, failed(String) }` + `player`/`audioPairs`/`selectedPair`/`audibleGroup`. `@MainActor`-isolated.
- `Services/PlaybackCoordinator.swift` ~295 LOC — `@MainActor @Observable`. Observes `ScanModel.selectedClipID` via `withObservationTracking` with re-arm. `currentPrepTask: Task<Void, Never>?` for supersedence. Pipeline: ensure HTTP server up → cache lookup → if miss: prepareOutputDir → spawn transcode via injected closure → consume events until firstSegmentReady (yield rest to `Task.detached` tail-drain) → build playlist URL (`master.m3u8` for ≥2 pairs, `playlist.m3u8` for ≤1) → `replaceCurrentItem` or new `AVPlayer` (with `automaticallyWaitsToMinimizeStalling = false`) → observe `AVPlayerItem.status` → flip to `.playing` or `.failed`.

- **Refactor in ScanModel:** `selectedClipID` lifted from ContentView's `@State` onto the model. Required for `withObservationTracking` since it observes properties on the @Observable, not @State. Inside ContentView's body, the new pattern is `@Bindable var model = model` shadowing — Apple's documented way to project Bindings out of an @Observable held in @State.

### Wave P8.4 — clean ffmpeg cleanup on Cmd-Q (commit pending)

Researched macOS-specific patterns first via apple-docs MCP + web search. Key findings:

- **No Darwin equivalent of Linux `PR_SET_PDEATHSIG`.** Force-quit / SIGKILL of the parent reparents children to launchd; cannot be defended against without PTY/session-leader hacks that aren't worth it for one ffmpeg child. **Document and accept.**
- `applicationShouldTerminate(_:)` returning `.terminateLater` is Apple's only documented async-cleanup hook. AppKit spins the runloop in `NSModalPanelRunLoopMode` until you call `reply(toApplicationShouldTerminate: true)` — `@MainActor` Tasks AND `Process.terminationHandler` callbacks fire normally during the wait. **Synchronous-block-on-quit is wrong** (freezes the runloop, beachballs, misses terminationHandler queue posts).
- ffmpeg's SIGTERM handler flushes `EXT-X-ENDLIST` in 10–50 ms (per spike P1.2). 500 ms grace = 10× headroom. Plan's original 2 s figure was over-conservative.
- `Process.terminate()` is async-signal-only (sends SIGTERM, returns immediately). `waitUntilExit()` blocks. For coordinator-side cleanup we don't need the Process object; raw `kill(pid_t, Int32)` from `Darwin` works once we track pids.

Implementation:
- `Services/AppDelegate.swift` ~30 LOC — `@MainActor final class AppDelegate: NSObject, NSApplicationDelegate`. `applicationShouldTerminate` returns `.terminateLater`, kicks `Task { @MainActor in await coordinator.shutdown(); NSApp.reply(toApplicationShouldTerminate: true) }`. Holds `weak var coordinator: PlaybackCoordinator?` set by the App struct after init.
- `AvidMXFPeekApp.swift` — `@NSApplicationDelegateAdaptor(AppDelegate.self) var appDelegate`; in `initializeCoordinatorIfNeeded` after `coordinator = c`, also `appDelegate.coordinator = c`. Order matters: coordinator owned by `@State` in the App struct, delegate weak-refs it.
- `PlaybackCoordinator.swift` — added `import Darwin`, `private var activePIDs: Set<pid_t> = []`. `.started(pid)` event captures into a local `trackedPID` and inserts into the set. `.completed`/`.failed` (both main loop and tail-drain) remove the pid. New `func shutdown() async` SIGTERMs all tracked pids, sleeps 500 ms, SIGKILLs. New `fileprivate func removeActivePID(_:)` for the tail-drain `Task.detached` to call back through `await self?.removeActivePID(pid)` (since the detached task isn't @MainActor-isolated).
- pbxproj — 4 edits registering AppDelegate.swift.
- New test `shutdownSIGTERMsTrackedSubprocess` — spawns a real `/bin/sleep 30`, registers its pid via the stub transcode closure's `.started(pid:)`, calls `coordinator.shutdown()`, polls for `sleepProc.isRunning == false` over up to 1 s. Real-subprocess test, not a mock — cleanest way to verify actual signal delivery.

Smoke test: launched app, killed via `killall AvidMXFPeek`, verified `pgrep -f "ffmpeg|AvidMXFPeek"` returns nothing. Clean shutdown confirmed.

## Decisions logged

- **§11.1–§11.4 plan corrections** committed in `docs/plans/2026-04-22-player-hls.md` before writing P6 code. Pattern: research before implementation, append corrections as numbered sections (matches §10 precedent).
- **TranscoderCall closure typealias over factory or protocol.** Smallest possible test injection seam; no new files, no new protocols, just a behavioral signature.
- **`selectedClipID` lifted to ScanModel.** Required for `withObservationTracking` semantics. The `@Bindable var model = model` body shadowing pattern bridges this back to ContentView's existing Binding-style usage.
- **`.terminateLater` over synchronous block.** Apple-blessed; lets Tasks fire during the wait. Synchronous block would peg the runloop and miss `terminationHandler` posts.
- **No `POSIX_SPAWN_SETSID` or process-group tricks for v1.** `.terminateLater` covers Cmd-Q (~99% of quits). SIGKILL of the app itself is unrecoverable; documented limitation.
- **500 ms SIGTERM grace, not 2 s.** 10× empirically-measured ffmpeg ENDLIST flush time.

## Notes / lessons

- **SourceKit lies during multi-file additions.** Entire waves of "cannot find type" errors appeared every time new orphan files existed before pbxproj wiring. `xcodebuild` is the source of truth — it cleared every false alarm. Running an actual compile after each wave (before chasing language-server errors) saved a lot of false-debugging time.
- **`@Bindable var model = model` shadowing inside `body`** is the canonical Apple pattern for projecting Bindings out of an `@Observable` held in `@State`. Initially tried `Bindable(model).selectedClipID` (looks plausible) — fails with "Missing argument label 'projectedValue:'". The wrapper-as-property declaration is required.
- **AVPlayerView vs SwiftUI.VideoPlayer matters for editorial.** `VideoPlayer` is one-liner-easy but loses 8+ controls that an editor expects (timecodes, frame-stepping, JKL, beginTrimming, etc.). For a Pro-level audit tool, `NSViewRepresentable(AVPlayerView)` is non-optional — frame-accuracy is the whole reason an editor opens a preview at all.
- **Plan corrections as numbered sections, not inline edits.** `§10` (post-P1) and `§11` (pre-P6) are now the second-most-read parts of the plan after the wave checklists. Future-me reading this in six months can see the design evolution. Inline edits would have erased the journey.
- **Real subprocess in shutdown test, not a mock.** `kill(real_pid, SIGTERM)` and `sleepProc.isRunning` give ground truth. Mocking `kill()` would have tested our own implementation, not the OS contract.
