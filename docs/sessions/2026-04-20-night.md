## Gotchas captured for future-me

- **Real-data validation is not optional for speculative parsers.** The Wave 2 `MXFInfoXMLParser` was written against bmx XML documentation screenshots and educated guesses; it was ~0% correct against actual output. The Wave 5.2 gate existed to catch exactly this, and it did â€” but the cost was building a whole parser + tests that got deleted in one day. Next time: grab ONE real sample during Wave 2 planning even if the full validation suite waits until Wave 5. Ten minutes of early sampling saves hours of speculative parser work.
- **Output-style modes aren't hard rules** â€” per this session's collaboration-style feedback memory (written earlier in the evening when I scaffolded a `fatalError` TODO for the `consume` loop), follow the style's *philosophy* only when it matches the actual task shape. The mechanical `for await` loop wasn't a "meaningful 5-10 lines of code shape the design" opportunity; the real design had been settled by A1+B3. Same principle re-applied in this session: when user approved the pivot plan, I executed straight through without inventing fork points.
- **PBXFileSystemSynchronizedRootGroup is the right abstraction for test folders.** New test `.swift` files drop in without pbxproj edits. Same abstraction *used to* wire up the `Resources/lib` dylib folder for the bmx toolchain â€” which meant I had to delete the root group + exception set when cleaning it up in 4.6-A. Conceptually the same feature; two different use sites in this project, both now retired except the test folder.
- **sed on pbxproj for bulk file deletion works when each reference contains the filename in a comment.** `sed -i '' "/filename.swift/d" project.pbxproj` removes all four references (PBXBuildFile / PBXFileReference / PBXGroup child / PBXSourcesBuildPhase files) at once because each of those sections formats the reference as `ID /* filename.swift in Sources */` or similar. One sed per dead file; 30 files â†’ 30 sed commands â†’ 120 pbxproj lines cleanly removed. Would fail if pbxproj IDs were commented with non-filename markers or if the Edit section conflicted, but those cases don't exist in a well-formed Xcode-generated pbxproj.
- **Write tests whose failure mode is obvious at a glance.** The CRLF identity test cost me one iteration because `csv.components(separatedBy:).count - 1` vs. `csv.filter { $0 == "\n" }.count` involves a subtle off-by-one reasoning that isn't worth the risk. "Strip X, assert Y doesn't remain" is always clearer.
- **Test target via Xcode click > manual pbxproj surgery.** 8 new PBX sections + dependency wiring + scheme logic is real work and brittle to get right. User's 10-second Fileâ†’Newâ†’Target dialog did it correctly. Manual pbxproj surgery is worth it for *maintenance* edits (line-level removal of known entries) but not for *creation* of new structural sections.
- **Debug `.app` sizes fluctuate with test-harness presence.** Going from pure `build` to `test`-mode build inflated the .app from 78 MB â†’ 105 MB because DerivedData carried test product overhead. `clean build` returned 74 MB, the real distributable shape. Measure clean-built sizes when you want stable numbers.

## Cookbook candidates from this session

Three patterns stand out as potentially reusable, but per the output-style feedback memory I won't capture them reflexively:

1. **`readDataToEndOfFile + waitUntilExit` over `readabilityHandler` for short-lived subprocesses** â€” avoids tail-byte race that the old pattern has at larger pipe volumes. Applies any time you're shelling out to get a small discrete result rather than streaming progress. Simpler code, no handler cleanup, no race window.
2. **Mass dead-code sweep recipe** (sed on pbxproj + disk delete + trim-in-place for partially-dead files + test suite as safety net) â€” exactly what 4.6-B executed. Reusable process shape for any "inherited project carries 70% of its parent's code as dead weight" situation.
