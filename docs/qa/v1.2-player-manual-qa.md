# v1.2 Player — Manual QA Checklist

**For:** Wave §P7.4 (functional QA) + §P8.3 (memory profiling)
**App build under test:** Debug `.app` from `~/Library/Developer/Xcode/DerivedData/AvidMXFPeek-*/Build/Products/Debug/AvidMXFPeek.app`
**Test data root:** any `Avid MediaFiles/MXF/1` folder (or parent `MediaFiles/MXF/`)
**Expected baseline (per spike P1.5):** ~2.5 s clip-input → first-frame on a clean cache miss

## Setup

- [ ] Clean build product (rebuild Debug from Xcode or `xcodebuild` to be sure)
- [ ] Cache wipe before first run: `rm -rf ~/Library/Caches/com.lucesumbrarum.AvidMXFPeek/previews/`
- [ ] At least one **YouTube-import clip** in the folder (e.g. Kowloon, Dresser, Fookls — anything tall/long-form)
- [ ] At least one **camera-native clip** (4-8 audio stems if possible — for multi-pair coverage)
- [ ] One known-orphan clip (optional, just to confirm scan still works)

## §P7.4 — Functional QA (golden path)

### Launch + folder pick

- [ ] App launches without crash, no `[AvidMXFPeek] playback init failed` in Console
- [ ] Cache directory exists post-launch: `ls ~/Library/Caches/com.lucesumbrarum.AvidMXFPeek/previews/`
- [ ] Cmd-O opens the folder picker; pick MXF folder
- [ ] Scan completes; clips appear in the table

### First clip — cache miss

- [ ] Toggle **Inspector ON** (top-right toolbar button)
- [ ] Click first clip in the table
- [ ] **Preparing spinner** appears with elapsed-seconds counter ticking
- [ ] **First frame visible within ~2.5 s** (note actual time: ___ s)
- [ ] **Timecode is visible** in the AVPlayerView controls (HH:MM:SS:FF format)
- [ ] **Frame-stepping buttons** are visible/usable
- [ ] JKL keyboard shortcuts work (J = back, K = pause, L = forward; Space = play/pause)
- [ ] Audio is audible

### Multi-pair clip

- [ ] Click a clip with **≥4 audio stems**
- [ ] **Audio pair Picker** appears above the player (popup-style)
- [ ] Picker shows pair labels like `A01_A02`, `A03_A04`
- [ ] Switch pair → audio swaps, **playback continues at currentTime** (not reset to start)
- [ ] Re-switch back to first pair → no retranscode, instant swap
- [ ] **No new ffmpeg process** appears in `ps aux | grep ffmpeg` during the switch

### Mono / odd-count fallback

- [ ] Click a clip with **odd audio stem count** (3, 5, 7…)
- [ ] Picker shows last entry as `*_mono` (label suffix)
- [ ] Audio plays — both channels carry the same content (mono-routed)

### Video-only clip

- [ ] Click a clip with **0 audio stems**
- [ ] Picker is **hidden** (not just disabled — should not be in the layout)
- [ ] Video plays silent — no error banner

### Cache hit

- [ ] Click a previously-played clip again
- [ ] **No preparing spinner** — player jumps straight to `.playing`
- [ ] First frame visible **<500 ms**
- [ ] No new entry under the cache root

### Scrubbing

- [ ] Drag the AVPlayerView scrubber forward and back
- [ ] Playback is reactive (no multi-second stalls except possibly past the transcode horizon on first-play of a long clip — see §10.5 risk note)
- [ ] Scrubbing past the transcoded extent on a first-play long clip: AVPlayer either stalls silently until the segment lands, or recovers — no crash

### Inspector toggle teardown

- [ ] With a clip playing, toggle **Inspector OFF**
- [ ] Audio stops within ~100 ms (player paused, item replaced with nil)
- [ ] `ps aux | grep ffmpeg` shows no in-flight ffmpeg
- [ ] Toggle Inspector ON again → "No clip selected" placeholder (selection was cleared)

### Rapid clip selection

- [ ] Click 5 different clips in <2 s
- [ ] Final clip's preview appears
- [ ] `ps aux | grep ffmpeg` shows **at most 1 ffmpeg** (others terminated)
- [ ] Watch Console for `[AvidMXFPeek]` errors during the burst

### Failure paths

- [ ] **Disk-full simulation** (optional, advanced): mount a tmpfs that's <1 GB, point cache root via launch arg if supported, click a clip → red banner "Disk preflight failed: …"
- [ ] **Corrupted MXF** (use the `corruptMXF` test fixtures from `MXFFolderScannerTests` if you have similar real data) — should land as parse error in scan, no preview attempt
- [ ] **Network-mounted MXF** (SMB/AFP if available) — confirm scan + preview still work; record any latency hits

### Quit cleanup (P8.4)

- [ ] Click a clip, wait until preparing spinner appears
- [ ] **While transcoding**, Cmd-Q
- [ ] App quits within ~1 s
- [ ] `ps aux | grep ffmpeg` shows **no orphan ffmpeg** owned by you
- [ ] Cache directory still has the (incomplete) entry — its `.transcode-state` JSON should be `failed` or `running` (not `complete`)

## §P8.3 — Memory profiling (Instruments)

### Setup

- [ ] Open Instruments → choose **Allocations** template
- [ ] Attach to running AvidMXFPeek build (or launch via Instruments)
- [ ] Filter to AvidMXFPeek process

### Scenarios — record each as a separate trace

#### Scenario A: long-clip scrubbing

- [ ] Pick a 20-30+ minute clip
- [ ] Let preview start playing
- [ ] Scrub forward in 30-second jumps for 5 minutes wall time
- [ ] **Watch for:**
  - [ ] CVPixelBuffer allocations should stay roughly bounded (not strictly monotonic-up)
  - [ ] Persistent bytes <500 MB (rough; the goal is "doesn't grow without bound", not a hard cap)
  - [ ] No leaks of `AVPlayerItem`, `AVPlayer`, `AVURLAsset`

#### Scenario B: rapid clip switching

- [ ] Click through 20 different clips, ~5 s pause each
- [ ] **Watch for:**
  - [ ] AVPlayerItems are released after each switch (`replaceCurrentItem(with: newItem)` should drop the old one)
  - [ ] No `PreviewTranscoder` or `Process` leaks
  - [ ] HTTP server connections close (NWConnection should not accumulate)

#### Scenario C: cache-hit churn

- [ ] Click 5 cached clips in a tight loop, 50 times
- [ ] **Watch for:**
  - [ ] Allocations should plateau, not grow
  - [ ] No `URL` / `String` leaks from the playlist URL construction

### Pass criteria (lenient — this is v1, not Pro Tools)

- [ ] No graph trending strictly up after 10 minutes of mixed use
- [ ] Persistent bytes <800 MB during sustained playback
- [ ] No findings flagged by **Leaks** instrument
